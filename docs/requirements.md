# requirements

大洲市児童館クラブ予約システムの機能要件・非機能要件をまとめる。
CLAUDE.md に書かれた固定要件を「真実のソース」とし、本ファイルは詳細・補足を扱う。
矛盾が出た場合は CLAUDE.md を優先する。

---

## 1. 用語

| 用語 | 意味 |
| --- | --- |
| 施設 / 館 | 大洲児童館、喜多児童館、徳森児童センターの3つ |
| クラブ | 各館が開催するイベント単位の予約対象 |
| 予約 | 特定クラブに対する利用者からの申込み |
| 予約待ち | 定員超過時に積み上げられるキュー |
| 繰り上げ | キャンセル発生時に予約待ち先頭が確定状態に変わること |
| admin | 管理者 |
| super_admin | 全館（非削除）の権限を持つ admin。アカウント追加・館の管理が可能 |

## 2. 施設コード（DB / URL / 予約番号の prefix に共通）

館は DB の `facilities` テーブルで動的に管理する（ADR-0018）。初期データは
以下の 3 件を seed する。`code` は予約番号の prefix・URL パラメータに使われる
ため、新館を追加する際は **小文字英数字のみ** で命名する。

| 館名 | コード |
| --- | --- |
| 大洲児童館 | `ozu` |
| 喜多児童館 | `kita` |
| 徳森児童センター | `toku` |

新規館の追加・既存館の編集・ソフト削除は super_admin が `/admin/facilities`
から行える。

## 3. 利用者機能

### 3.1 クラブ一覧
- **並び順**: 日付降順 → 時間降順
- **カラム**: 日付 / 時間 / 館名 / クラブ名 / 定員 / 予約数 / 予約する / 写真
- **状態表示**: 「空きあり」「キャンセル待ち」「終了」（DB enum は `waitlisted` のまま、UI ラベルのみ）
  - 「終了」の判定は、クラブ開始日時を過ぎたとき
  - 受付終了後も 1年間は一覧に残す
- **写真**: 有効な外部リンク（http/https）があれば「見る」、なければ「準備中」
- **対象年齢**: 一覧カードまたは詳細で表示

### 3.2 予約入力
| 項目 | 必須 | 備考 |
| --- | --- | --- |
| 保護者（複数可） |  | 任意。最大 10 名。入力した行は氏名 + ふりがな（ひらがなのみ、サーバー側検証）|
| お子さま（複数可） | ✓ | 1 名以上 10 名以下。各人ごとに氏名 + ふりがな（ひらがなのみ、サーバー側検証）|
| 電話番号 | ✓ | 国内番号形式（ハイフン可）|
| メールアドレス | ✓ | RFC 5322 準拠の簡易検証 + MX 確認は任意 |
| 備考 | | 500字以内 |

DB 上は `reservation_parents` / `reservation_children` に position 付きで正規化して保存する（ADR-0005 の更新版）。
利用者向けメールは汎用文面で送信し、保護者氏名による個別の挨拶は行わない（自動送信メール
であり、宛先メールアドレスから本人宛と判断できるため）。

### 3.3 確認画面
- 入力内容を表示
- 利用規約を併記
  - 予約は先着順であり、結果は登録メールに送る
  - キャンセルは 2営業日前の17時まで
  - それ以降のキャンセルおよび無断欠席は原則禁止
  - キャンセルが続く場合は今後の利用を断ることがある
- 「予約を確定する」で送信

### 3.4 完了画面とメール
- 完了画面には予約番号と、予約確認画面の URL（クリック可能なリンク）を案内
- 予約番号
  - 形式: `ozu_123456` / `kita_123456` / `toku_123456`
  - 各館の `prefix` は `facilities.code` で動的に決まり、新館の追加で増やせる
  - 全体ユニーク、再利用しない
  - 6桁数字部分は人間が読めるよう **推測困難すぎない桁数** に抑える
  - セキュリティ担保のため内部に別の **secure token（32文字以上の乱数）** を持ち、予約確認 URL で secure token を検証する
- メール本文（テキスト版 + HTML 版を multipart で送信、ADR-0016）
  - 先頭に「このメールは予約システムから自動送信しています。」の固定注記
  - 予約完了 / 予約待ち（何番目か）の区別
  - 予約番号 / クラブ名 / 開催日時の表
  - 予約確認・キャンセル用 URL（HTML 版では緑のボタン + 同じ URL の文字列を併記）
  - キャンセル期限の説明（「開催日の 2 営業日前 17 時まで」固定文言）
  - フッターに全館の連絡先（非削除の `facilities.phone` を列挙）

### 3.5 予約確認・キャンセル画面
- 予約番号を入力して内容確認（または メールリンクから直接）
- 現在の定員、予約数を表示
- キャンセル操作でキャンセル確認メールを送信
- 予約待ち状態でもキャンセル可能
- キャンセル期限を過ぎた予約は、画面上でキャンセル不可である旨を明示

### 3.6 キャンセル発生時
- 予約数を即時デクリメント
- 予約待ち先頭がいれば自動で確定状態に変更し、対象者に予約完了メール送信

### 3.7 データ保持
- クラブ開始日時から 1年間は DB・一覧ともに保持（「終了」表示）
- 1年以上経過したクラブと紐づく予約情報は削除対象

## 4. 管理者機能

### 4.1 ログイン
- ID（またはメール）/ パスワード でログイン
- 認証情報とセッションは server-side enforcement
- パスワードはハッシュ保存（bcrypt / argon2、または Supabase Auth 任せ）

### 4.2 画面構成
- ログイン後画面に以下のエントリ
  - クラブ一覧（担当館のみ）/ クラブを新規登録
  - 事業マスター（`/admin/programs`、全 admin が閲覧・編集可能）
  - パスワード変更（右上）
- super_admin のみ表示されるメニュー
  - **アカウントの追加**（`/admin/accounts`、招待メール送信・館権限の付与）
  - **館の管理**（`/admin/facilities`、館の CRUD・電話番号の編集・予約番号 prefix 設定）

### 4.3 クラブ登録
- 館選択は自分に紐づく館のみ
- 入力: 日付、開始時刻、終了時刻、クラブ名、定員、対象年齢、写真URL（任意）、説明など
- 登録後は即時予約可能
- 写真 URL は http(s) のみ許可。ドメイン allowlist の要否は open-questions.md で検討

### 4.4 クラブ一覧・編集
- 自分の館のクラブのみ閲覧・編集可能
- 編集操作は audit_logs に記録

### 4.5 予約者一覧・管理者キャンセル（ADR-0021）
- 各クラブの予約者一覧（`/admin/clubs/[id]/reservations`）から、active な予約に対して
  「キャンセルする」リンクを表示
- 押下で `/admin/reservations/[id]/cancel` の確認画面に遷移し、予約内容と「利用者へ
  キャンセル通知メールが送信されること」「元が予約完了の場合は予約待ち先頭が繰り上がること」
  「取り消し不可」を明示してから確定する
- 確定後は利用者経路と同じキャンセルメール / 繰り上げメールを送信
- 締切（2 営業日前 17 時）の判定は管理者経路では行わない（締切後の無連絡欠席や
  日程変更時の強制キャンセルが主な運用想定）
- `reservation.admin_cancel` を audit_logs に記録（個人情報は metadata に含めない）

### 4.6 アカウント追加（super_admin のみ）
- email / 一時パスワード or 招待リンク方式
- 館ごとの権限を付与
- super_admin 権限を付与する場合は全3館への権限付与が必要
- 監査ログに記録

### 4.7 パスワード変更
- 現パスワード確認 + 新パスワードの複雑性チェック
- Supabase Auth を使う場合はその機能に乗る

## 5. 非機能要件

### 5.1 セキュリティ
- server-side authorization 必須（フロントでの表示制御は UX のみ）
- 同時予約の定員超過を防ぐためトランザクション必須
- 個人情報をログに出さない
- secrets をコミットしない
- 外部リンクは http/https のみ許可
- CSRF / XSS / SQLi 観点でレビュー

### 5.2 性能 / 可用性
- 愛媛県大洲市の児童館利用者規模を想定（同時接続 <100 相当）
- Vercel + Supabase の無料または低価格プランで成立することを目標
- ピーク時は人気クラブの予約開始直後のみ

### 5.3 UI / UX
- 大洲市の子育てサイトに馴染むやさしいトーン
- 保護者が迷わない導線
- エラーメッセージは責めない表現
- スマホ必須

### 5.4 アクセシビリティ
- WCAG 2.1 AA を目標（色コントラスト、キーボード操作、ARIA）

### 5.5 運用
- 管理操作の監査ログ
- retention cleanup（1年経過クラブ削除）は Supabase cron もしくは Vercel Cron

## 6. 受け入れ条件サマリ

| 観点 | 条件 |
| --- | --- |
| 予約 | 定員超過しない。同時予約でも DB で整合 |
| 予約番号 | 人間可読 / グローバルユニーク / 再利用なし |
| キャンセル | 期限内のみ。期限後は UI で不可。予約待ち解消で通知 |
| 権限 | 自館以外のデータには一切アクセスできない（API 直叩きでも） |
| 個人情報 | ログ・console・error reporting に漏れない |
| モバイル | 320px 幅から崩れない |
| 保持 | 終了後 1年 表示、1年超は削除 |
