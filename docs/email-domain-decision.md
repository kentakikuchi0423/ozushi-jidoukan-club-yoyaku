# メール送信用ドメインの決定 — 市担当者との協議資料

最終更新: 2026-04-25

## 1. 背景と目的

大洲市児童館クラブ予約システム（以下「本システム」）から保護者へ送信する
メール（予約確認・キャンセル確認・キャンセル待ち繰り上がり等）の **送信元
ドメイン** をどう確保するかを決める必要がある。

現状（試験導入直前）は、メール配信サービス Resend が用意する共有ドメイン
`onboarding@resend.dev` を使っている。これは **Resend 登録者本人にしか
メールが届かない** 仕様であり、外部の保護者（試験対象を含む）に予約確認
メールが届かない。**外部送信を行うには独自ドメインの DNS で SPF / DKIM /
DMARC を検証する必要がある**。

## 2. 何を決めるのか（協議の最終アウトプット）

以下 4 つの問いに、市担当者と合意した答えを出す。

| # | 問い | 想定される答え |
|---|---|---|
| 1 | 送信元ドメインに **市が既に保有する** ドメインを使うか、新規取得するか | 既存利用 / 新規取得 |
| 2 | 既存利用なら、**サブドメインの委譲または DNS レコード追加** が可能か | 可能 / 不可（理由） |
| 3 | 新規取得なら、**所有者は市か、保守事業者か** | 市所有 / 事業者所有 |
| 4 | 試験期間中の **暫定送信元** をどうするか | 個別許可リスト / 仮ドメイン / 暫定無し |

## 3. 論点

### 3-1. 信頼性（最重要）

- 保護者は **市役所からのメール** として受信する。送信元ドメインが市と無関係
  だと、迷惑メール扱いされたり、保護者が「詐欺メールでは？」と警戒する。
- Gmail / docomo / au / softbank などの大手メール事業者は、**ドメインの
  評判スコア** で受信可否を判定する。**新規取得した独自ドメイン**でも数週間〜
  数ヶ月で評判が形成される。**無料ドメイン (`.tk` `.ml` 等) は Gmail/キャリア
  で機械的にスパム判定** されるため、本システム用途では事実上使えない。

### 3-2. 法・規程

- 大洲市の情報セキュリティポリシー、個人情報保護条例の対象範囲。
- 子どもの氏名・年齢・連絡先メールアドレスを扱うため、**個人情報の取り扱い** に
  ついて市の所管課と整合させる必要がある（送信内容にも氏名・予約番号が含まれる）。
- 市の **公式ドメイン (`*.lg.jp` `*.city.<地名>.<県名>.jp`) を第三者運用の
  Web サービスから使う場合、自治体側のドメインポリシーで禁じられているケース
  がある**（なりすまし防止のため）。市の情報政策担当に事前確認が必要。

### 3-3. 運用の継続性

- 担当者の異動やシステムベンダーの変更があっても、**ドメインの更新（年次
  料金支払い・DNS 編集）が止まらない** 体制が必要。
- 所有者が個人や事業者の場合、**廃業・退職時にドメインが失効** すると、
  メールどころか過去メールに含まれた予約確認 URL もすべて消滅する。
- 結論: **本格運用フェーズでは、ドメインは市または市の指定する組織が保有する** のが望ましい。

### 3-4. DNS 編集権限と作業速度

- DNS レコード（SPF/DKIM/DMARC）の追加・更新には **DNS 管理画面への編集
  権限** が必要。
- 自治体の DNS は **総務系・情報政策担当** が管理していることが多く、
  外部の事業者が直接編集できないケースが大半。**1 件のレコード追加に申請
  → 承認 → 反映で数日〜数週間** かかる。
- DKIM 鍵は **再発行（ローテーション）** が定期的に発生する。年 1〜2 回の
  申請が必要になる前提で運用フローを決める。

### 3-5. 試験導入期間中の暫定対応

- 試験対象が **保護者数名（5 人以下）** なら、Resend の **Allowed
  Recipients** にメアドを個別登録することで `onboarding@resend.dev` のまま
  送信可能（無料・即時）。
- **6 人を超える** ならドメイン検証が現実的に必要。
- 試験中だけ仮の独自ドメインで運用 → 本番で市ドメインに切り替える場合、
  **過去のメールに含まれた予約確認 URL が失効** することに注意（試験中の
  予約データが本格運用に持ち越される設計だと混乱する）。

## 4. 仕組みの解説 — メールが届く / 届かないの境目

「市のドメインを使う」が技術的に何を意味するのか、市担当者と話す前提を
共通言語化するための節。ITC 専門でなくても理解できるよう、図で説明する。

### 4-1. 大前提: ドメインは持ち主しか DNS を書き換えられない

`city.ozu.ehime.jp` の DNS（ドメイン情報を答えるサーバ）を運用しているのは
大洲市である。インターネット上で「`city.ozu.ehime.jp` についての情報を教
えてほしい」という質問は、最終的に市の DNS サーバに行き着き、そこに登録さ
れた値だけが正解として返る。

つまり、**市の許可なく `city.ozu.ehime.jp` 配下に何かを設定することは不可能**。
他人が勝手に「メール送信用の設定を追加します」と言っても、世界中のどの DNS
サーバもその設定を返さない。

### 4-2. もし市の許可なく `city.ozu.ehime.jp` を送信元にしたら何が起きるか

「設定を追加する」のはダメでも、メールの送信元（From 欄）に
`noreply@city.ozu.ehime.jp` と **書く** こと自体は可能。ただし、現代の
メール基盤はそれをほぼ確実に弾く仕組みになっている。

受信側のメール事業者（Gmail / docomo / au / softbank など）が必ず行う 3 つの検証:

#### 検証 1. SPF — 「この IP からの送信は許可されてるの？」

受信側は `city.ozu.ehime.jp` の SPF レコード（市が公開する送信許可リスト）を
読む。Resend のサーバ IP が許可リストに **無い** ので、検証失敗。

#### 検証 2. DKIM — 「ドメインの秘密鍵で署名されてるの？」

DKIM は「正規の送信者だけが持つ秘密鍵で、本文に電子署名を付ける」仕組み。
`city.ozu.ehime.jp` の秘密鍵は **市の所有物** で、Resend には渡されていない
ので署名できない。検証失敗。

#### 検証 3. DMARC — 「SPF/DKIM が失敗したらどうする？」

DMARC は **ドメインの持ち主が** 「もし上の 2 つが失敗したら、受信側は破棄
してね / 迷惑メール扱いしてね / 通すけど報告だけ送ってね」を宣言する仕組み。

市が `p=reject`（厳格）を設定していれば、SPF・DKIM が通らないメールは
受信サーバが **完全に破棄**。`p=quarantine` なら迷惑メールフォルダ送り。

#### つまり

```
[送信] noreply@city.ozu.ehime.jp から保護者の Gmail へ送信
          ↓
[Gmail 側の検証]
  ・SPF:   city.ozu.ehime.jp の許可 IP リストに Resend は無い → ✗
  ・DKIM:  city.ozu.ehime.jp の秘密鍵で署名されていない        → ✗
  ・DMARC: 市の DMARC 設定が p=reject → 即座に破棄
          ↓
[結果] 保護者には永久に届かない
       送信元（Resend）にエラーバウンスが返る
       市側にも「不正なメールが自ドメインを名乗って送信されている」という DMARC レポートが届く
```

→ **市側にとっては「自ドメインを騙ったなりすまし送信」として観測される** ため、
無断使用は技術的に成立しないだけでなく、運用面でも問題化する。

### 4-3. 正しく使うために市が DNS に追加するレコード

市が「このシステムからのメールは正規のものとして扱ってよい」と DNS で
宣言する。Resend がコピペ用の値を生成し、市はそれを DNS 管理画面に登録
するだけ。

#### 例: `yoyaku.city.ozu.ehime.jp` をメール送信用に使う場合

| 種別 | ホスト名 | 値 | 役割 |
|---|---|---|---|
| **TXT** | `yoyaku.city.ozu.ehime.jp` | `v=spf1 include:_spf.resend.com ~all` | SPF: Resend からの送信を許可 |
| **TXT** | `resend._domainkey.yoyaku.city.ozu.ehime.jp` | `p=MIGfMA0GCSqGSIb3DQEBA...`（200〜400 文字の公開鍵） | DKIM: Resend の署名を検証する公開鍵 |
| **MX** | `send.yoyaku.city.ozu.ehime.jp` | `feedback-smtp.resend.com`（優先度 10） | バウンス受信用 |
| **TXT** | `_dmarc.yoyaku.city.ozu.ehime.jp` | `v=DMARC1; p=none; rua=mailto:...` | DMARC: 検証失敗時の扱い + 監視レポート受信先 |

#### このレコード追加で「何が分岐するか」を一行ずつ説明

- **SPF**: 受信側が「Resend の IP 範囲を市が許可している」と確認できる
- **DKIM**: 受信側が公開鍵で署名検証 → 「市の秘密鍵で署名されている = 改ざんなし」
- **MX**: 配信不能のメールが「送信失敗」として戻る場所
- **DMARC**: 監視レポートが市側に届くようになり、なりすまし攻撃を検知可能

レコード追加後、世界中の DNS サーバに反映するまで **数分〜数時間**（TTL 次第）。
反映後、Resend 側の管理画面で各レコードが「Verified」に切り替わる → 送信開始可能。

#### レコード追加の作業時間そのものは

DNS 管理画面にログインしてレコードを 4 件貼り付けて保存するだけ。
**実作業は 5〜10 分**。後述の作業フロー（誰が、どんな手続きで、にかかる時間）
と分けて考えること。

### 4-4. 「市の DNS 申請プロセス」の中身

DNS レコード追加自体は 5〜10 分の作業。**時間がかかるのは市役所内部の
事務手続き** で、現実によくあるパターンが 3 つある。

#### パターン A: 市が自前で DNS 管理（情報政策担当が直接編集）

```
本システム保守側 → 情報政策担当へ依頼（メール / 文書）
   「以下 4 件の DNS レコードを追加してください」（添付: Resend 表示の値）
   ↓
情報政策担当者が DNS 管理画面（cPanel / Route53 等）にログイン
   ↓
レコードを 4 件追加・保存
   ↓
完了通知
```

**所要日数の目安**: 1〜3 営業日。**追加作業費はゼロ**（職員作業のため）。

#### パターン B: 委託業者が DNS 管理（市の職員は直接編集できない）

```
本システム保守側 → 情報政策担当へ依頼
   ↓
情報政策担当 → 委託業者へ作業発注
   ↓
業者が作業 → 完了報告
   ↓
情報政策担当 → 保守側へ完了通知
```

**所要日数の目安**: 5〜14 営業日。**業者への作業費**（数千〜数万円）が
発生する場合があり、軽微変更扱いか別途見積もりかは契約次第。

#### パターン C: 自治体クラウド契約に紐づく / 契約変更が必要

```
情報政策担当へ依頼
   ↓
「自治体クラウドの契約範囲外」と判明
   ↓
契約変更の起案・決裁
   ↓
変更契約締結 → 業者作業 → 反映
```

**所要日数の目安**: 数週間〜数ヶ月。**変更契約料金** が発生する可能性。

#### つまり、市担当者との初回協議で最初に確認したいのは

> 「うちの市の `city.ozu.ehime.jp` の DNS は、誰が、どうやって編集していますか？」

ここで A / B / C のどれかが判明し、本格運用までのリードタイム見積もりが
立てられる。

## 5. 選択肢

### 選択肢 A: 市の既存ドメイン配下のサブドメインを使う（推奨度 ★★★）

例: `yoyaku.city.ozu.ehime.jp`、または `mail.yoyaku.city.ozu.ehime.jp`

| 項目 | 内容 |
|---|---|
| 取得コスト | 0 円（市の既存ドメイン配下） |
| 信頼性 | ◎ 市公式ドメインなので保護者に最も信頼される |
| 立ち上げ速度 | △〜× 市の DNS 申請プロセスを通る必要があり、数日〜数週間 |
| 運用継続 | ◎ 市が管理 |
| 必要な調整 | 市情報政策担当と協議。サブドメインの委譲（NS 委譲） or 個別レコード追加（A/CNAME/TXT）の方式選択 |

### 選択肢 B: 市が新規ドメインを取得する（推奨度 ★★★）

例: `ozushi-jidoukan-yoyaku.jp` `ozushi-yoyaku.com` 等を **市名義で** 取得

| 項目 | 内容 |
|---|---|
| 取得コスト | 年 1,500〜2,500 円程度（`.jp` は少し高め） |
| 信頼性 | ○ 市名義の SOA 情報が公開され、相応に信頼される |
| 立ち上げ速度 | ◎ 1〜2 日で完了可能 |
| 運用継続 | ◎ 市が管理 |
| 必要な調整 | 市の予算化、登録業者選定（お名前.com / Value Domain / Cloudflare 等）、DNS 編集権限の所在 |

### 選択肢 C: 保守事業者が新規ドメインを取得 → 後日市へ譲渡（推奨度 ★★）

| 項目 | 内容 |
|---|---|
| 取得コスト | 年 1,500〜2,500 円（一旦事業者負担、譲渡時に精算） |
| 信頼性 | ○（譲渡前後で同じ） |
| 立ち上げ速度 | ◎ 即日〜翌日 |
| 運用継続 | △ 譲渡完了までは事業者依存。譲渡手続きにレジストラ毎の事務（数日〜2 週間） |
| 必要な調整 | 譲渡時期と譲渡条件の事前合意（できれば書面）、譲渡後の DNS 編集体制 |

### 選択肢 D: 試験中は Allowed Recipients、本格運用は A or B（推奨度 ★★）

試験対象が 5 人以下に限定できる場合の **段階的アプローチ**:

1. 試験期間（〜数週間）: `onboarding@resend.dev` のまま、Resend の Allowed Recipients に対象保護者のメアドを登録
2. 本格運用前に A または B でドメイン検証
3. 切替日に Vercel 環境変数 `RESEND_FROM_ADDRESS` を本ドメイン宛に変更 → Redeploy

| 項目 | 内容 |
|---|---|
| 取得コスト | 試験中 0 円 / 本格運用は A または B のコスト |
| 信頼性 | 試験中: 限定的（送信元が `resend.dev`） / 本格運用: 高 |
| 立ち上げ速度 | 試験中: 即時 / 本格運用: A or B 次第 |
| 運用継続 | A or B に準ずる |
| 必要な調整 | 試験対象の人数管理、本格運用前の本ドメイン準備 |

## 6. 推奨

**最終形は選択肢 A（市既存ドメインのサブドメイン）または B（市新規取得）。**
これは「市が情報発信の主体である」点から信頼性・継続性のいずれにおいても優位
だから。

**試験導入期間中は選択肢 D（Allowed Recipients での暫定運用）** を採用し、
**並行して市と A / B の協議を進める** のが現実的。試験対象を 5 人以下に
絞り、本格運用切替日に向けて DNS 設定を準備する。

## 7. 市担当者との協議で確認すべき事項

協議の場で **市側に答えを出していただきたい質問** を以下にまとめる。

### 6-1. ドメインの保有・管理体制

1. 市が現在保有しているドメインは何か（`city.ozu.ehime.jp` 等）。それぞれの **管理者・所管課** はどこか。
2. それらドメインの DNS レコード追加・編集は **どの部署が、どのプロセスで** 行うか（紙申請 / メール / チケット）。所要日数の目安は。
3. **サブドメインの委譲（NS 委譲）** は前例があるか。可能か。
4. メール送信用に **新規ドメインを市名義で取得** する場合の手続きは（予算・契約・登録業者選定）。

### 6-2. ポリシーと制約

5. 市の情報セキュリティポリシーで、**送信元ドメインに関する制約** はあるか（例: なりすまし防止のため許可されたシステムからのみ `*.city.ozu.ehime.jp` から送信可、など）。
6. **個人情報保護条例** に基づく、外部 SaaS（Resend / Supabase / Vercel）の利用について審査・承認のプロセスは。
7. **DMARC ポリシー** を市ドメイン全体に設定済みか（`p=reject` / `p=quarantine` / `p=none`）。設定済みの場合、本システムの送信が DMARC を満たすよう SPF/DKIM の追加が必要。

### 6-3. 運用継続性

8. ドメインの **更新料・DNS 管理料金** の予算枠はどこが持つか（情報政策費 / 児童館事業費 / 別の科目）。
9. **担当者異動** に備えた引き継ぎ体制（複数名でのドメイン管理権限保有、文書化）の有無。
10. **保守事業者の変更** が発生した場合のドメインの扱い（次の事業者へ DNS 編集権限を引き継ぐ手順）。

### 6-4. 試験導入期間中の取り扱い

11. 試験期間（想定: 〇月〇日〜〇月〇日）の **対象保護者数** は何名か。Allowed Recipients での暫定運用の許容範囲（5 人以下）に収まるか。
12. 試験中に作成された予約データ（テスト含む）を、**本格運用にどう引き継ぐか**（クリアして本番開始 / 試験データを残す）。

## 8. 想定タイムライン（選択肢 A or B 採用時）

```
[Day 0]   協議開始 / 上記質問に対する市の方針確認
[Day 1-3] 市側で内部確認・予算確認（B の場合）
[Day 5-10] ドメイン取得 or サブドメイン委譲手続き
[Day 10-12] DNS レコード（SPF/DKIM/DMARC）追加
[Day 12-14] Resend 側で検証完了 → Vercel 環境変数更新 → Redeploy
[Day 14+] 本ドメインから保護者宛にテスト送信、Gmail / キャリアメールで受信確認
[Day 15-]  本格運用開始（試験データの整理を経て）
```

選択肢 A のサブドメイン委譲が市の DNS 体制次第で長引く場合は、その期間
中に **選択肢 D（Allowed Recipients）** で試験を継続する。

## 9. 技術側で並行して進められること

協議中、保守事業者（Claude Code 含む）側で先に進められる作業:

- [ ] Resend Account Settings の **Allowed Recipients** に試験対象保護者のメアドを登録（試験開始前に必須）
- [ ] 本ドメイン決定後、Resend Domains に **登録するためのレコード** を控える（SPF/DKIM/DMARC のテンプレ）
- [ ] DKIM 鍵の長さ（**2048 bit を推奨**、1024 bit は将来的に弱体化）
- [ ] DMARC レポート受信用メアドの用意（最初は `p=none`、運用後に `quarantine` へ段階引き上げ）
- [ ] 切替日のオペレーション手順書（`docs/operations.md` への追記）

## 10. 備考

- 本システムは Vercel で動作しており、**サイト URL（`*.vercel.app` または独自ドメイン）と
  メール送信元ドメインは別個に決められる**。送信元ドメインだけ市ドメイン、
  サイト URL は Vercel のままで OK。
- ただし、**サイト URL も市ドメインにそろえた方が利用者にとって直感的**。
  本格運用時には `yoyaku.city.ozu.ehime.jp` 等への移行も合わせて検討する。
