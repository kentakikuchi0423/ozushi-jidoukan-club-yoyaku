## 3. 第 1 部 — メール送信用ドメイン

### 3-1. 背景と決めるべきこと

本システムから保護者に送るメール（予約確認・キャンセル確認・キャンセル待ち繰り上がり）の **送信元ドメイン** を、本格運用ではどうするか。

試験導入中は保守側が取得した `ozu-system.com` から送信中。本格運用では信頼性・継続性の観点から **市ドメイン** に切り替えるのが望ましい。

決めるべき問い:

| # | 問い | 想定される答え |
|---|---|---|
| 1 | 送信元ドメインに **市が既に保有する** ドメインを使うか、新規取得するか | 既存利用 / 新規取得 |
| 2 | 既存利用なら、**サブドメインの委譲または DNS レコード追加** が可能か | 可能 / 不可（理由） |
| 3 | 新規取得なら、**所有者は市か、保守事業者か** | 市所有 / 事業者所有 |

### 3-2. 仕組みの解説 — メールが届く / 届かないの境目

「市のドメインを使う」が技術的に何を意味するのかを共通言語化するための節。
ITC 専門でなくても理解できるよう図で説明する。

#### 3-2-1. 大前提: ドメインは持ち主しか DNS を書き換えられない

`city.ozu.ehime.jp` の DNS（ドメイン情報を答えるサーバ）を運用しているのは大洲市。インターネット上で「`city.ozu.ehime.jp` についての情報を教えてほしい」という質問は、最終的に市の DNS サーバに行き着き、そこに登録された値だけが正解として返る。

つまり、**市の許可なく `city.ozu.ehime.jp` 配下に何かを設定することは不可能**。

#### 3-2-2. もし市の許可なく市ドメインを送信元にしたら何が起きるか

メールの送信元（From 欄）に `noreply@city.ozu.ehime.jp` と **書く** こと自体は可能。ただし、現代のメール基盤はそれをほぼ確実に弾く仕組みになっている。

受信側のメール事業者（Gmail / docomo / au / softbank など）が必ず行う 3 つの検証:

- **検証 1. SPF**: 送信元 IP が、市が公開する送信許可リストに含まれているか → 含まれていなければ NG
- **検証 2. DKIM**: 市の秘密鍵で本文が署名されているか → されていなければ NG
- **検証 3. DMARC**: 上の 2 つが NG だった時、ドメイン所有者が「破棄しろ」と宣言していれば即座に破棄

```
[送信] noreply@city.ozu.ehime.jp から保護者の Gmail へ
          ↓
[Gmail 側の検証]
  ・SPF:   許可 IP リストに無い → ✗
  ・DKIM:  秘密鍵で署名されていない → ✗
  ・DMARC: p=reject → 即破棄
          ↓
[結果] 保護者には届かない / 市側には「なりすまし」レポートが届く
```

→ **無断使用は技術的にも運用上も成立しない**。市と協調して DNS にレコードを追加する必要がある。

#### 3-2-3. 正しく使うために市が DNS に追加するレコード

市が「このシステムからのメールは正規のものとして扱ってよい」と DNS で宣言する。Resend がコピペ用の値を生成するので、市はそれを DNS 管理画面に登録するだけ。

例: `yoyaku.city.ozu.ehime.jp` をメール送信用に使う場合の登録レコード（4 件）:

| 種別 | ホスト名 | 値 | 役割 |
|---|---|---|---|
| TXT | `yoyaku.city.ozu.ehime.jp` | `v=spf1 include:_spf.resend.com ~all` | SPF: Resend からの送信を許可 |
| TXT | `resend._domainkey.yoyaku.city.ozu.ehime.jp` | DKIM 公開鍵（200〜400 文字） | DKIM: 署名検証用の公開鍵 |
| MX | `send.yoyaku.city.ozu.ehime.jp` | `feedback-smtp.resend.com`（優先度 10） | バウンス受信 |
| TXT | `_dmarc.yoyaku.city.ozu.ehime.jp` | `v=DMARC1; p=none;` | 検証失敗時の扱い + 監視レポート |

DNS 管理画面にログインしてレコードを 4 件貼り付けて保存するだけで、**実作業は 5〜10 分**。

#### 3-2-4. 「市の DNS 申請プロセス」の中身

DNS レコード追加自体は 5〜10 分。**時間がかかるのは市役所内部の事務手続き** で、現実によくある 3 パターン:

| パターン | 内容 | 所要日数の目安 | 追加費用 |
|---|---|---|---|
| **A** | 市が自前で DNS 管理（情報政策担当が直接編集） | 1〜3 営業日 | 0 円 |
| **B** | 委託業者が DNS 管理（市職員は直接編集できない） | 5〜14 営業日 | 数千〜数万円（軽微変更扱いか別見積もりか） |
| **C** | 自治体クラウド契約に紐づく / 契約変更が必要 | 数週間〜数ヶ月 | 変更契約料金が発生する可能性 |

→ **市担当者との初回協議で最初に確認したいのは**:

> 「うちの市の `city.ozu.ehime.jp` の DNS は、誰が、どうやって編集していますか？」

A / B / C のどれかが判明すれば、本格運用までのリードタイム見積もりが立つ。

### 3-3. 選択肢 A〜D

#### 選択肢 A: 市の既存ドメイン配下のサブドメインを使う ★★★

例: `yoyaku.city.ozu.ehime.jp`、または `mail.yoyaku.city.ozu.ehime.jp`

| 項目 | 内容 |
|---|---|
| 取得コスト | 0 円（市の既存ドメイン配下） |
| 信頼性 | ◎ 市公式ドメインなので保護者に最も信頼される |
| 立ち上げ速度 | △〜× 市の DNS 申請プロセスを通る必要があり、数日〜数週間 |
| 運用継続 | ◎ 市が管理 |
| 必要な調整 | 市情報政策担当と協議。サブドメインの委譲（NS 委譲） or 個別レコード追加（TXT/MX）の方式選択 |

#### 選択肢 B: 市が新規ドメインを取得する ★★★

例: `ozushi-jidoukan-yoyaku.jp` `ozushi-yoyaku.com` 等を **市名義で** 取得

| 項目 | 内容 |
|---|---|
| 取得コスト | 年 1,500〜2,500 円程度（`.jp` は少し高い） |
| 信頼性 | ○ 市名義の SOA 情報が公開され、相応に信頼される |
| 立ち上げ速度 | ◎ 1〜2 日で完了可能 |
| 運用継続 | ◎ 市が管理 |
| 必要な調整 | 市の予算化、登録業者選定、DNS 編集権限の所在 |

#### 選択肢 C: 保守事業者が新規ドメインを取得 → 後日市へ譲渡 ★★

| 項目 | 内容 |
|---|---|
| 取得コスト | 年 1,500〜2,500 円（一旦事業者負担、譲渡時に精算） |
| 信頼性 | ○（譲渡前後で同じ） |
| 立ち上げ速度 | ◎ 即日〜翌日 |
| 運用継続 | △ 譲渡完了までは事業者依存。譲渡手続きにレジストラ毎の事務（数日〜2 週間） |
| 必要な調整 | 譲渡時期と譲渡条件の事前合意（できれば書面）、譲渡後の DNS 編集体制 |

> **試験導入中は実質これに該当**: 保守側が `ozu-system.com` を取得して運用中。

#### 選択肢 D: 試験中は Allowed Recipients、本格運用は A or B ★★

試験対象が 5 人以下に限定できる場合の **段階的アプローチ**:

1. 試験期間: `onboarding@resend.dev` のまま、Resend の Allowed Recipients に対象保護者のメアドを登録
2. 本格運用前に A または B でドメイン検証
3. 切替日に Vercel 環境変数 `RESEND_FROM_ADDRESS` を本ドメイン宛に変更 → Redeploy

> **本プロジェクトでは選択肢 C（独自ドメイン取得）を選んだため、現状は適用外**。本格運用時の選択肢として比較対照に残す。

### 3-4. 推奨

**最終形は選択肢 A（市既存ドメインのサブドメイン）または B（市新規取得）**。「市が情報発信の主体である」点から信頼性・継続性のいずれにおいても優位だから。

**試験導入期間中は選択肢 C（保守事業者が `ozu-system.com` を運用中）** を採用済み。並行して市と A / B の協議を進める。本格運用切替日に向けて、市側で DNS 設定の準備を進めてもらう。

### 3-5. 市担当者へ確認すべき事項（ドメイン関連）

#### ドメインの保有・管理体制

1. 市が現在保有しているドメインは何か（`city.ozu.ehime.jp` 等）。それぞれの **管理者・所管課** はどこか
2. それらドメインの DNS レコード追加・編集は **どの部署が、どのプロセスで** 行うか（紙申請 / メール / チケット）。所要日数の目安は
3. **サブドメインの委譲（NS 委譲）** は前例があるか。可能か
4. メール送信用に **新規ドメインを市名義で取得** する場合の手続きは（予算・契約・登録業者選定）

#### ポリシーと制約

5. 市の情報セキュリティポリシーで、**送信元ドメインに関する制約** はあるか
6. **個人情報保護条例** に基づく、外部 SaaS（Resend / Supabase / Vercel）の利用について審査・承認のプロセスは
7. **DMARC ポリシー** を市ドメイン全体に設定済みか（`p=reject` / `p=quarantine` / `p=none`）。設定済みの場合、本システムの送信が DMARC を満たすよう SPF/DKIM の追加が必要

#### 運用継続性

8. ドメインの **更新料・DNS 管理料金** の予算枠はどこが持つか（情報政策費 / 児童館事業費 / 別の科目）
9. **担当者異動** に備えた引き継ぎ体制（複数名でのドメイン管理権限保有、文書化）の有無
10. **保守事業者の変更** が発生した場合のドメインの扱い

### 4-2. 各サービスの料金体系と閾値

#### 4-2-1. Vercel — Web サーバ

| プラン | 月額 | 主な上限 |
|---|---|---|
| **Hobby（Free）** | 無料 | 帯域 100 GB/月、サーバ実行 100 GB-時間/月、Cron Job 1 個 |
| **Pro** | $20/月（約 ¥3,200） | 帯域 1 TB/月、Cron Job 40 個、業務利用 OK |

**児童館規模で実際に超えるか？**

- 想定: 月 600 予約（3 館 × 10 クラブ × 20 人）→ ページビュー 6,000〜10,000 回
- 帯域消費: 1 ページ 200 KB として **月 2 GB 程度** → 100 GB の **2%** 以下
- **技術的には Free 枠内で十分動く**

ただし **Vercel Hobby プランは利用規約上「個人の非商用利用」限定**。市の業務利用は厳密には規約違反扱いとなるため、**本格運用では Pro プラン推奨**。

→ **Vercel Pro: 年 ¥38,400**

#### 4-2-2. Supabase — データベース + 認証

| プラン | 月額 | 主な上限 |
|---|---|---|
| **Free** | 無料 | DB 500 MB、帯域 5 GB/月、認証 50,000 MAU、**1 週間アクセスなしでプロジェクト一時停止**、バックアップ 7 日 |
| **Pro** | $25/月（約 ¥4,000） | DB 8 GB、帯域 250 GB/月、100,000 MAU、自動停止なし、日次バックアップ + Point-in-Time Recovery |

**児童館規模で実際に超えるか？**

- 1 予約 ≈ 1 KB × 月 600 件 = **月 0.6 MB**、年間でも **7 MB 程度**
- 500 MB に達するには **約 70 年分** の予約が必要
- **Free 枠内に十分収まる**

ただし Free プランの「1 週間アクセスなしで一時停止」が運用上のリスク:

- 夏休み・年末年始など利用が極端に少ない時期に偶然 1 週間アクセスがゼロになると、**翌週いきなり保護者が予約しようとした際にサイトが応答しない** 事故が起こりえる
- 復旧は Supabase Studio から手動 Resume → 5〜10 分
- 完全に防ぐには **Pro プラン** が必要

→ **Supabase Pro: 年 ¥48,000**

#### 4-2-3. Resend — メール送信

| プラン | 月額 | 主な上限 |
|---|---|---|
| **Free** | 無料 | **3,000 通/月**、**100 通/日**、検証ドメイン 1 個 |

**児童館規模で実際に超えるか？**

- 1 予約あたり最大 2 通（確定 + キャンセル時）
- 月 600 予約 × 2 = **月 1,200 通** → Free 枠 3,000 の **40%**

→ **Resend Pro: 年 ¥38,400**

## 7. 備考

### 7-1. サイト URL とメール送信元の関係

- 本システムは Vercel で動作しており、**サイト URL（`*.vercel.app` または独自ドメイン）と
  メール送信元ドメインは別個に決められる**
- 送信元ドメインだけ市ドメイン、サイト URL は Vercel のままでも技術的には問題ない
- ただし、サイト URL も市ドメインにそろえた方が利用者にとって直感的。本格運用時には `yoyaku.city.ozu.ehime.jp` 等への移行も検討する

### 7-2. ベンダーロックインについて（参考）

協議で「特定サービスへの依存度」を聞かれた場合の回答素材:

- データは標準 PostgreSQL なので、Supabase 以外への移行は技術的に可能
- メールは Resend 以外（SendGrid / Mailgun / SES 等）に切替可能
- ホスティングも Next.js なので Vercel 以外（Cloudflare Pages / 自前 VPS）に切替可能
- → ロックインは弱く、乗り換えコストは小さい

### 7-3. 試験中のデータの取り扱い

- 試験中に作成された予約データ（テスト含む）を、**本格運用にどう引き継ぐか** は協議事項
- 選択肢: ①クリアして本番開始 / ②試験データを残してそのまま継続
- メールの予約確認 URL は **secure_token 付き** で生成されているため、URL を持っている保護者は試験データへもアクセスできる。本格運用切替時は試験データを残すか消すかを明確に決める
