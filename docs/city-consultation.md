## 4. 第 1 部 — メールの差出人ドメイン

### 4-1. 何を決めるのか

本システムから保護者に送るメール（予約確認・キャンセル・繰り上げ）の **差出人ドメイン** を本格運用ではどうするか。

| # | 問い | 想定される答え |
|---|---|---|
| 1 | 差出人ドメインに市の既存ドメインを使うか、新規取得するか | 既存利用 / 新規取得 |
| 2 | 既存利用なら、サブドメインを切る方式が可能か | 可能 / 不可（理由） |

### 4-2. なぜ「市の許可なし」では使えないのか

メールの差出人欄に `noreply@city.ozu.ehime.jp` と書くこと自体は可能ですが、現代のメール基盤は概ねこれを弾きます。

受信側（Gmail / docomo / au / softbank など）が行う検証:

- **SPF**: 送信元が市の許可リストに含まれているか
- **DKIM**: 市の秘密鍵で本文が署名されているか
- **DMARC**: 上 2 つが NG だった時、ドメイン所有者の宣言に従って破棄

```
市の許可なし → SPF ✗ / DKIM ✗ → DMARC で破棄 → 保護者には届かない
```

→ 市と協調して **市の DNS に「Resend からの送信を許可する」レコード** を追加する必要があります。

### 4-3. 設定の全体像 — 誰が何をどこに

市のドメインから保護者宛にメールを送るまでの作業を時系列で整理:

| ステップ | 作業 | 場所 | 作業者 | 時間 |
|---|---|---|---|---|
| 1 | 使いたいドメインを Resend に登録 | Resend | 保守事業者 | 5 分 |
| 2 | Resend が DNS レコード（4 件: SPF / DKIM / MX / DMARC）の値を発行 | Resend | （自動） | 即時 |
| 3 | **発行されたレコードを市の DNS に追加してもらう** | **市の DNS** | **市の DNS 管理者** | 設定 5〜10 分／調整に数日〜数週間 |
| 4 | DNS 反映後、Resend で「認証完了」を確認 | Resend | 保守事業者 | 数分〜数時間 |
| 5 | Vercel の環境変数 `RESEND_FROM_ADDRESS` を更新 | Vercel | 保守事業者 | 5 分 |
| 6 | Vercel を再デプロイ | Vercel | 保守事業者 | 数分 |
| 7 | テスト送信、Gmail / キャリアメールで受信確認 | — | 保守事業者 + 児童館 | 30 分〜半日 |

> **時間がかかるのはステップ 3 だけ**。それ以外は数分〜数時間で完了します。

#### 市が DNS に追加するレコードの例

`yoyaku.city.ozu.ehime.jp` を使う場合:

| 種別 | ホスト名 | 値 | 役割 |
|---|---|---|---|
| TXT | `yoyaku.city.ozu.ehime.jp` | `v=spf1 include:_spf.resend.com ~all` | SPF |
| TXT | `resend._domainkey.yoyaku...` | DKIM 公開鍵 | DKIM |
| MX | `send.yoyaku...` | `feedback-smtp.resend.com` 優先度 10 | バウンス受信 |
| TXT | `_dmarc.yoyaku...` | `v=DMARC1; p=none;` | DMARC |

実際の値は Resend がコピペ用に発行するので、市側はそれを DNS 管理画面に登録するだけです。

### 4-4. 市の DNS 管理者は誰か / どんな対応が必要か

市のドメイン（例: `city.ozu.ehime.jp`）の DNS は、市内部または委託業者が管理しています。**どのパターンに該当するか** によって、必要な対応が変わります。

| パターン | 状況 | 必要な対応 |
|---|---|---|
| **A. 市が自前で管理** | 情報政策担当が DNS 管理画面に直接ログインできる | 情報政策担当者が、Resend の発行レコード（4 件）を DNS 管理画面にコピペで追加・保存。庁内で完結する |
| **B. 委託業者が管理** | 市職員は直接編集できず、業者に依頼する形 | 市から委託業者に **作業依頼書を発行**。「軽微変更扱いか / 別見積もりが必要か」を事前確認。依頼ルートと作業実施スケジュールを業者と調整 |
| **C. 自治体クラウド契約に紐づく** | DNS が自治体クラウドサービスの一部として運用されている | クラウド運用業者に **DNS 変更が契約範囲内かを確認**。契約変更や追加見積もりが必要なケースもある。関係部署（情報政策・契約担当・運用業者）の調整が必要 |

協議の最初に確認したい問い:

> **市の DNS は、誰が、どうやって編集していますか？**

A / B / C のどれかが分かれば、必要な手続きと対応者が決まります。

> **なお**: 試験用ドメイン `ozu-system.com` を管理している Cloudflare Registrar は保守事業者の登録口で、**市の既存ドメインが Cloudflare で管理されているとは限りません**。市の DNS は別途確認が必要です。

### 4-5. 市が既に使っているメール送信サービスがある場合

市が既に SendGrid / Amazon SES / Postmark / Brevo / Microsoft 365 / Google Workspace 等のメール配信サービスを使っている場合、**Resend の代わりに流用できる可能性** があります。具体的な代替候補と切替工数は **§7** を参照してください。

ただし「使えるかどうか」は次の確認が必要です:

- 自動送信メールを送ってよい契約・運用ルールか
- API または SMTP で接続できるか
- `noreply@…` のような自動送信用アドレスを発行できるか
- 月 1,000〜2,000 通の送信上限を満たすか
- 個人情報を含むメールを扱う運用上の問題はないか

**現実的な落としどころ**: まずは現在の Resend を継続したまま差出人ドメインだけ市のサブドメインに切替える方式が最も簡単です。既存サービスへの移行は安定運用後の選択肢として検討します。

### 4-6. 推奨

**最終形**: 市既存ドメインのサブドメインを切る（例: `yoyaku.city.ozu.ehime.jp`）

| 観点 | 内容 |
|---|---|
| 取得コスト | **0 円**（市の既存ドメイン配下） |
| 信頼性 | 市公式ドメインで保護者に最も信頼される |
| 運用継続 | 市が管理 |
| 必要な調整 | 市 DNS 管理者と協議。サブドメイン委譲（NS 委譲）または DNS レコード追加（TXT/MX）の方式選択 |

**試験中は保守側の `ozu-system.com` を継続**。市側 DNS 設定が完了次第、Vercel 環境変数を切替えるだけで本ドメインに移行できます。

## 7. 参考: 市の既存 IT 基盤での代替可能性

市が既に持っている IT 基盤で、本システムの一部または全部を **SaaS から市側基盤に置き換える** 選択肢があります。

### 「市が X を使っているなら → Y で代替できる」早見表

各ケースで **市側 / 保守側がそれぞれ何をする必要があるか** を整理しました。

| 市側で使っているもの | 代替の方向性 | 必要な対応 |
|---|---|---|
| **AWS（クラウド契約）** | Vercel → AWS Amplify Hosting / ECS Fargate / Elastic Beanstalk | 市側: 保守事業者向けの IAM ロール / アクセス権限付与。保守側: Next.js デプロイ実装、CI/CD 構築、cron の代替（EventBridge 等） |
| **Azure（クラウド契約）** | Vercel → Azure App Service | 市側: Azure サブスクリプションへのアクセス権限付与。保守側: App Service へのデプロイ実装と CI/CD 構築 |
| **Google Cloud（クラウド契約）** | Vercel → Cloud Run | 市側: GCP プロジェクトへのアクセス権限付与。保守側: Docker 化と Cloud Run デプロイ、Cloud Scheduler で cron 代替 |
| **自治体クラウド / Linux VPS** | Vercel → 同基盤に Node.js デプロイ | 市側: SSH アクセスとポート開放（80/443）。保守側: Node.js + Nginx + Let's Encrypt + systemd + cron の構築 |
| **PostgreSQL サーバ（オンプレ or マネージド）** | Supabase の DB 部分 → 既存 PostgreSQL（認証は別実装が必要） | 市側: DB 接続情報と RLS 用ロールの提供、ネットワーク疎通の許可。保守側: DB マイグレーション、Auth.js 等で認証フロー書き換え |
| **Microsoft 365** | Resend → M365 SMTP リレー / Supabase Auth → Entra ID | 市側: SMTP 送信ライセンスの確認、Entra ID へのアプリ登録（OAuth クライアント）。保守側: メール送信ライブラリを SMTP に書き換え or Entra ID 認証統合の実装 |
| **Google Workspace** | Resend → Workspace SMTP リレー / Supabase Auth → Workspace SSO | 市側: SMTP リレーの設定許可、SSO アプリ登録。保守側: メール送信ライブラリを SMTP に書き換え or Workspace SSO 統合の実装 |
| **SendGrid / Amazon SES / Postmark / Brevo** | Resend → 既存メール配信サービス | 市側: API キーの発行、市ドメインに対する SPF / DKIM の値の差し替え（Resend 用 → 移行先用）。保守側: メール送信ライブラリのアダプタ差し替え |

### 代替できないもの

| 市側で使っているもの | 理由 |
|---|---|
| **Microsoft SQL Server / Oracle / MySQL** | 本システムの DB は PostgreSQL 前提。SQL 方言・行単位権限制御の方式が大きく異なるため大幅書き換えが必要（数週間〜） |
| **WordPress / 既存の市 Web サイト CMS** | アプリ種類が違うため Web ホストとしては不可 |
| **静的ホスティング基盤のみ** | Server Action / 動的ページ処理が必須なので不可 |
| **Microsoft 365 / Google Workspace（DB 用途）** | これらは DB 機能を持たない（メール・文書管理用） |

### 早期に代替検討が必要なケース

- **個人情報を市管理外のクラウドに置けない** 等の制約がある場合 → Supabase の代替（自治体クラウドや市の既存 PostgreSQL）を検討
- 市が **既に M365 / Google Workspace を導入** している場合 → メール送信や管理者認証で活用可能性がある

### 推奨

**最初は SaaS 構成（Vercel + Supabase + Resend）で本格運用を始め、運用が安定した後で代替検討するのが現実的** です。代替実装の工数（数日〜数週間）が本格運用開始時期を後ろ倒しにする上、児童館規模では SaaS の年額（〜10 万円）が許容範囲に収まるため。

## 8. 備考

### 8-1. サイト URL とメール差出人ドメインの関係

サイト URL（`*.vercel.app` または独自ドメイン）とメール差出人ドメインは別個に決められます。本格運用が安定した後、サイト URL も市ドメイン（例: `yoyaku.city.ozu.ehime.jp`）にそろえる移行を検討します。

### 8-2. ベンダーロックインについて

- データは標準 PostgreSQL なので Supabase 以外への移行は技術的に可能（§7）
- メールは Resend 以外（SendGrid / Postmark / SES 等）に切替可能（§7）
- ホスティングも Next.js なので Vercel 以外（AWS / Azure 等）に切替可能（§7）
- → ロックインは弱く、乗り換えコストは比較的小さい

## 9. 市担当者への確認事項

協議の場でそのまま使える質問リスト。優先度順に整理:

### 9-1. 必ず確認したいこと

1. 市のドメイン（例: `city.ozu.ehime.jp`）の **DNS は誰がどう管理しているか**
2. **サブドメインの委譲（NS 委譲）** の前例はあるか / 可能か
3. 市が既に **メール配信サービス**（SendGrid / SES / M365 / Workspace 等）を使っているか
4. 市が **クラウド契約**（AWS / Azure / GCP / 自治体クラウド）を持っているか
5. **個人情報を含むメール** を外部 SaaS 経由で送ることに、運用上・契約上の問題はないか
6. 個人情報保護条例に基づく **外部 SaaS（Resend / Supabase / Vercel）の審査・承認プロセス** はどうなっているか

### 9-2. 本格運用前までに決めること

7. 本格運用時の **差出人メールアドレス**（例: `noreply@yoyaku.city.ozu.ehime.jp`）
8. ドメイン更新料・年間維持費の **予算枠** の所管（情報政策費 / 児童館事業費 等）
9. **DMARC ポリシー** が市ドメイン全体に設定されているか（設定済みなら本システムの送信が満たすよう調整必要）

### 9-3. 任意（後追いでも可）

10. 本格運用時の **予約サイト URL** も市ドメインにするか
11. **担当者異動** に備えた引き継ぎ体制（複数名でのドメイン管理権限、文書化）
12. **保守事業者の変更** が発生した場合のドメイン・サービスの扱い

## 付録 A. 各サービスの詳細料金（参考）

§6 で示した想定構成の根拠資料。各サービスの料金体系と上限・実際のトラフィック想定を載せます。

#### A-1. Vercel（Web サーバ）

| プラン | 月額 | 主な上限 |
|---|---|---|
| Hobby（Free） | 無料 | 帯域 100 GB/月、Cron Job 1 個、**個人の非商用利用限定** |
| Pro | $20/月（約 ¥3,200） | 帯域 1 TB/月、Cron Job 40 個、業務利用 OK |

**児童館規模での実消費**: 月 600 予約 → 帯域 月 2 GB（Hobby 上限の 2%）。**機能上は Hobby で十分動くが、組織利用は規約上 Pro 推奨**。

#### A-2. Supabase（DB + 認証）

| プラン | 月額 | 主な上限 |
|---|---|---|
| Free | 無料 | DB 500 MB、認証 50,000 MAU、**1 週間アクセスなしで自動停止**、バックアップ 7 日 |
| Pro | $25/月（約 ¥4,000） | DB 8 GB、自動停止なし、PITR + 日次バックアップ |

**児童館規模での実消費**: 1 予約 ≈ 1 KB × 月 600 件 = 月 0.6 MB。500 MB に達するには約 70 年分必要。Free 枠で十分だが、自動停止が運用リスク（夏休みなど 1 週間利用ゼロが起きうる時期に注意）。

#### A-3. Resend（メール送信）

| プラン | 月額 | 主な上限 |
|---|---|---|
| Free | 無料 | 3,000 通/月、**100 通/日**、検証ドメイン 1 個 |
| Pro | $20/月（約 ¥3,200） | 50,000 通/月、1,000 通/日、検証ドメイン 10 個 |

**児童館規模での実消費**: 月 1,200 通（予約確定 + キャンセル）→ Free 枠の 40%。ただし夏休み前等の予約一斉受付で 1 日 100 通超に達する可能性があるため、その時期に Pro 化を検討。
