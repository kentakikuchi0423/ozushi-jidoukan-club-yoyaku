# 大洲市児童館クラブ予約システム

## プロジェクト概要
このリポジトリは、愛媛県大洲市の以下3施設のクラブ予約システムを構築するためのものです。

- 大洲児童館
- 喜多児童館
- 徳森児童センター

利用者はブラウザからクラブを予約でき、管理者は権限に応じてクラブ登録・編集・管理者アカウント管理を行います。

このプロジェクトでは、Claude Code を主体として、計画・実装・テスト・ドキュメント更新・Git運用まで一貫して進めます。

---

## 現在の状態（必ず最初に読む）
- 現在地・次の一手は常に `STATUS.md` を正とする。
- タスク全体像と進捗率は `TASKS.md`。
- 設計判断は `docs/decisions.md`、未確定事項は `docs/open-questions.md`。
- 開発は VS Code devcontainer 内で行う（`.devcontainer/devcontainer.json`）。

## コマンド

すべて devcontainer 内で実行する想定。`pnpm` は corepack 経由で有効化済み。

```bash
pnpm install            # 依存インストール
pnpm dev                # 開発サーバー (http://localhost:3000)
pnpm build              # 本番ビルド
pnpm start              # 本番サーバー (build 後)
pnpm lint               # ESLint (Next.js core-web-vitals + typescript + prettier)
pnpm typecheck          # tsc --noEmit
pnpm format             # Prettier で書き換え
pnpm format:check       # Prettier の差分確認のみ
pnpm test               # Vitest (単体・jsdom)
pnpm test:watch         # Vitest watch モード
pnpm test:e2e           # Playwright (build → start → テスト)
```

Playwright の初回だけブラウザバイナリが必要:

```bash
pnpm exec playwright install chromium
```

## アーキテクチャ

- `src/app/` — Next.js 15+ App Router のルート。`layout.tsx` が `lang="ja"` / Noto Sans JP を設定。`page.tsx` は Phase 1 段階のプレースホルダ。
- `src/lib/` — ドメインロジック（UI 非依存）。テストを隣接配置 (`*.test.ts`)。まずは `facility.ts`（3館コード/名前のマスタ）から。
- `src/server/` — server-only 処理（DB アクセス・Resend 送信・RPC 呼び出し）を置く予定（Phase 2 以降）。`SUPABASE_SECRET_KEY`（旧 service_role key 相当）はここからのみ参照する。
- `e2e/` — Playwright の E2E テスト。`playwright.config.ts` は `pnpm build && pnpm start` を起動してから実行する。
- `docs/` — 要件・アーキ・未確定事項・ADR・テスト方針・セキュリティレビューの真実のソース。
- `.env.example` を複製して `.env.local` を作る。実値はコミットしない。

技術スタックの採用理由は `docs/decisions.md`（ADR 形式）を参照。

## 最優先の原則
- いきなり大きく実装せず、まず探索し、設計し、小さく安全に進める。
- 会話履歴ではなく、リポジトリ内のファイルを真実のソースとする。
- 毎セッション開始時に `TASKS.md`、`STATUS.md`、`docs/open-questions.md`、`docs/decisions.md` を確認する。
- セッション終了前には、必ず `TASKS.md` と `STATUS.md` を更新する。
- 曖昧点は止まらずに合理的な仮説で前進しつつ、`docs/open-questions.md` に残す。
- 大きな変更の前に、必要な設計・判断を docs に残す。
- build / lint / typecheck / test を可能な限り常に通る状態に保つ。

## 固定要件
- クラブ一覧は **日付降順・時間降順**。
- 一覧には「日付」「時間」「館名」「クラブ名」「定員」「予約数」「予約する」「写真」を表示する。
- 館名は以下のみを扱う。
  - 大洲児童館
  - 喜多児童館
  - 徳森児童センター
- 予約状況は UI では「空きあり」「キャンセル待ち」「終了」（DB enum は `waitlisted` のまま）。
- 写真リンクが有効な場合のみ「見る」、ない場合は「準備中」と表示する。
- クラブには **対象年齢** を持たせる。
- クラブ登録後は即時予約可能。
- 受付終了後も **1年間はクラブ一覧に表示** する。
- **1年以上前のクラブと関連予約情報は削除対象** とする。

---

## 予約番号と予約処理
- 予約番号は人間が読める形式とする。
- 形式は以下。
  - `ozu_123456`
  - `kita_123456`
  - `toku_123456`
- 予約番号は **全体ユニーク** とし、再利用しない。
- ただし、キャンセルや照合の安全性のため、内部的には **別の secure token** を持つ。
- 予約確定・予約待ち判定は **必ずサーバー側** で行う。
- 同時予約でも定員超過しないよう、競合に耐える実装にする。
- キャンセル時は予約数を更新し、必要に応じて予約待ち先頭を繰り上げる。
- 繰り上げ時は対象利用者へ通知する。

---

## 認証・認可
- 管理者機能は ID/パスワードでログイン可能とする。
- 管理者は館ごとの閲覧・編集権限を持つ。
- 権限は必ず **server-side** で強制する。
- フロントエンドの表示制御だけに依存しない。
- 3館すべての権限を持つ管理者のみ、アカウント追加を許可する。
- 管理系変更には監査ログを残す前提で設計する。

---

## セキュリティ原則
- 個人情報を `console.log` やアプリログに出力しない。
- secrets、APIキー、実アカウント情報をコミットしない。
- 外部リンクは最低限の検証を行う。
- ユーザー入力は必ずサーバー側で検証する。
- 認可は必ずサーバー側で検証する。
- 危険な仮実装を放置しない。
- 公開リポジトリ前提で、漏えいしない構成を優先する。

---

## UI/UX 原則
- 大洲市の子育てサイトに馴染む、やさしく分かりやすいトーンを目指す。
- 利用者画面・管理者画面ともにスマホ対応を必須とする。
- 保護者が迷いにくい導線を優先する。
- エラーメッセージは責めない表現にする。
- 管理画面は派手さより、見やすさ・誤操作防止・保守性を優先する。

---

## 技術方針
- Next.js + TypeScript を前提に進める。
- UI は Tailwind CSS を基本とし、必要に応じて shadcn/ui を使う。
- 低コスト・保守性・実装速度を重視する。
- 認証、DB、メール送信、デプロイはベストプラクティスを優先して選定する。
- 初期実装では、変更しやすく、安全で、無料または低コストで始めやすい構成を優先する。

## Git 運用ルール
- 小さく意味のある単位で変更する。
- 小さく意味のある単位でコミットする。
- コミット前に差分を確認する。
- コミットメッセージは分かりやすく書く。
- 可能なら Conventional Commits を使う。
- `main` への直接的な破壊的作業は避ける。
- force push はしない。
- 履歴を書き換えない。
- git user.name / user.email が未設定なら推測しない。必要であることを明記する。
- remote 未設定なら、無理に push せず、必要手順を docs または STATUS に残す。

## 作業の進め方
各フェーズで次の順を守ること。

1. 現状把握
2. 関連 docs 更新
3. 実装
4. テスト
5. 進捗更新
6. 差分確認
7. 必要ならコミット

大きなフェーズに入る前には、完了条件を明確にすること。

## テスト方針
- 重要ロジックには自動テストを付ける。
- 予約・予約待ち・繰り上げ・権限は重点的にテストする。
- UI は必要に応じて E2E テストを作成する。
- コード修正時は関連テストを回す。
- 失敗時は原因と次アクションを `STATUS.md` に残す。

---

## Claude Code への期待動作
- 必要以上に質問せず、進められる範囲は自走する。
- ただし、外部依存、認証情報、破壊的判断、要件の根幹変更は勝手に確定しない。
- 長い手順や繰り返し作業は skills に切り出すことを検討する。
- 調査・実装・レビューを分けたい場合は subagents の利用を検討する。
- セッション再開時は、まず進捗ファイルを読み、現在地を把握してから着手する。

1. `TASKS.md`

## セッション終了時チェック
毎セッション終了前は以下を実施すること。

1. `TASKS.md` 更新
2. `STATUS.md` 更新
3. 必要な docs 更新
4. build / lint / typecheck / test の結果整理
5. 未解決事項の明記
6. 次にやることの明記
7. コミット可否の判断

---

## 禁止事項
- 個人情報のログ出力
- secrets のコミット
- 権限チェックなしの管理操作
- サーバー側検証なしの重要処理
- 破壊的変更の無記録実施
- docs を更新せずに大きな方針変更を進めること
